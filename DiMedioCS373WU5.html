Week 5 Write-Up

Very often malware includes code directed toward maintaining stealth and persistence within an infected system. One means of accomplishing this is by manipulating memory.

Rootkits, for example, conceal their existence and actions from the users and system processes. These programs reside in the computer's memory and from there, manipulate other areas of memory.

By way of review, the Windows operating system has a user mode and a kernel mode. The user mode includes the APIs for the operating system, user-mode drivers, and applications. This includes, for example, .exe and .dll files. On the other hand, kernel mode includes kernel-mode drivers, driver support routines, the operating system kernel, and file system drivers, all of which execute above the hardware abstraction layer situated over the hardware itself. Most rootkits are .sys files representing kernel-mode drivers.

In kernel memory, there is no security separation. Any kernel driver can access any part of memory. This includes many important data structures that are typically targeted by malware stealth techniques, including the System Service Descriptor Table (SSDT), I/O request packets (IRP), and Interrupt descriptor table (IDT), among others.

One example of a rootkit that uses stealth techniques directed at the SSDT is Agony, which, once executed, creates a hidden file on the disk and loads its code into memory.

Fig. 1 - Agony creates a file "wininit.sys" to store its malicious code on the disk. This file is hidden from view of the user and other system processes.

Tuluka (by Libertad) is a particularly useful tool in analyzing malware such as Agony. In this case, the program can be used to look at the SSDT/(SST) to determine whether the table has been modified. The program flags as suspicious any function for which the current pointer deviates from its original pointer, which here includes the functions NtEnumerateValueKey, NtQueryDirectoryFile, and NtQuerySystemInformation.

Another tool is livekd, which generally implements live kernel debugging. Here, we can use livekd to examine locations in memory and determine, for example, that the original location in memory that was in the SST for the function NtEnumerateValueKey does in fact contain the function.

Fig. 2 - Tuluka identifies three functions in the SST that have modified pointers.

Agony, having modified the SST to point to its own malicious code in memory instead of the actual function code, can now intercede whenever a program calls these functions. This process is known as hooking. For example, the files displayed by the Explorer program can be distorted, showing only a subset of what is actually on the disk. In this way, the malware hides evidence of its existence.

Another important concept here involves threads, processes, applications and the relationships between them. A thread is the smallest unit of execution. A process, on the other hand, can comprise multiple threads, and an application can comprise multiple processes and threads. It is important to understand the structure of threads in order to determine where the malicious code is hiding and operating from.

In general, executable code exists at different layers. The high-level language (HLL) that the programmer uses to write code is one layer of abstraction away from the assembly language, which is yet another layer above machine code, which is in hex format (e.g. as a series of hex values representing instructions and parameters). The machine code is loaded into RAM before being executed and then translated into a format to be consumed by the CPU. The CPU includes an instruction pipeline which can execute one instruction at a time, alternating between threads based on timing, scheduling, and priority. Typically, modern computers have CPUs with multiple cores, allowing each core to execute instructions from their own pipeline. As a result, many programs create multiple threads.

Each thread executing on a computer has its own user-mode stack in memory, which is used to store the thread's function calls and variables. Threads also have a kernel-mode stack, relevant to transferring control from the user to kernel.

Processes have their own memory, including object tables with handles to other objects known to the process, image areas which store the binary image of the executable code, and allocated heap/stack memory. Each discrete area of memory for a process has its own permissions (e.g. R, RX, RW) indicating whether the stored information can be read, written to, or executed.

Process Hacker is a helpful tool that gives users visibility of process memory for different processes executing on the machine. The program reveals these objects, which can provide clues to how a piece of malware operates.

In one example, a keylogger malware inserts code into memory that intercepts keystrokes from a keyboard. Here, the malware hides in an existing program (Notepad) by manipulating the memory of that program. Generally, this is known as process injection. Process Hacker can be used to look at Notepad's memory and determine if anything suspicious exists. In this case, a private page of memory (with no designated name, which is unusual) has read, write and execute permissions.

Fig. 3 - Process Hacker reveals a private memory page in the Notepad process with executable permissions.

One strategy for detecting rootkits relies on the fact that pointers (e.g. in the SSDT) need to point to areas of memory that are trusted and appropriate. For example, known APIs typically exist in a defined range in memory. Pointers to areas outside of this range can be flagged as suspicious.

Generally, you can follow suspicious pointers to find the malicious code. This code can then be extracted, analyzed and sent to a scanning service (e.g. for pattern matching), among other examples.

To this end, another helpful tool is Windbg, which offers kernel debugging.

In the previously mentioned Agony example, Windbg was used to follow the pointer in the SSDT to the malicious code by placing a breakpoint at the suspicious pointer.

Fig. 4 - Using Windbg, a breakpoint is strategically placed at the beginning of one of the malicious functions.

Next, Windbg can be used to step through the malicious code to identify its functions and delineate its bounds within the memory. Here, we are looking for the point in the code at which the malicious code returns to the origin, namely, the legitimate function that was hooked by the malware.

Fig. 5 - After stepping through the malicious code, the point at which the function returns to the original (legitimate) code is identified.

Windbg can even be used to patch the memory, restoring the SSDT to its original condition.

First, the SSDT is displayed, revealing the location in memory that stores (what should be) the pointer to the legitimate API function. We have this information from using Tuluka as previously described.

Fig. 6 - Windbg is used to identify the location in memory that needs to be patched.

Next, the memory itself is accessed using the Memory view of Windbg. This interface allows users to overwrite values stored in memory. Here, the original value is written in place of the malicious one.

Fig. 7 - Using the Memory view in Windbg, the SSDT is restored to its original state.

Tuluka can then be used to confirm that the patch was successful. In the Agony example, the NtEnumerateValueKey function (the one that was patched above) is no longer flagged as Suspicious.

Fig. 8 - Tuluka confirms that the memory patch was successful.

Agony represents one example of a rootkit memory manipulation strategy. Others include inline hooking, in which the first five bytes of the code in memory is modified , instead of the SSDT entry.

Additionally, malware can also deploy other stealth techniques, including bootkits. Bootkits rely on manipulation of the master boot record (MBR) in order to manipulate memory as the machine is booting up. Normally, in a preboot stage, the machine is powered on, and BIOS initializes the hardware and calls the MBR, which is code located at the start of the disk. In turn, during a boot stage, the MBR loads code from a bootsector of an active partition of the disk. The bootsector loads a bootloader from the file system, which in turn loads a configuration database and locates and loads the kernel for the operating system. Bootkits corrupt this process by modifying the MBR.

Other stealth techniques include file forging, memory forging, deploying watcher threads for self protection, attacking anti-virus programs, dissociating memory from its normally associated counterpart stored on the disk, removing file dependencies and manipulating trust mechanisms. In one example, the Brazilian Banker rootkit bypasses driver signing requirements in order to prevent security tools from loading. As a process for a security tool is about to start, the rootkit determines that it is security software and patches the memory so that it doesn't load.
