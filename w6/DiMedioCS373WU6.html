Week 6 Write-Up

Network security concerns creating a safe space within a network that keeps dangerous hosts and data out and prevents valuable data from being stolen. Generally, threats enter and exit via network connections.

Robustness Principal

The Robustness Principal is the notion that networks should be liberal in receiving traffic and conservative in sending traffic, with respect to different systems' specifications. In other words, a system should be able to process as many errors as can be forseen while sticking closely to prescribed protocols when sending outgoing traffic.

Reproduced below is an explanation of the Principal, with highlights indicating points of agreement or disagreement and bullet points providing commentary below.

Robustness Principle: 1980-1989 from RFC-1122 Jonathan Postel, 1989

Once there was a great man, named Postel.  See RFC 2468.

1.2.2 Robustness Principle

At every layer of the protocols, there is a general rule whose application can lead to enormous benefits in robustness and interoperability [ref to rfc760, 1980]:

                  “Be liberal in what you accept, and conservative in what you send”

Software should be written to deal with every conceivable error, no matter how unlikely; sooner or later a packet will come in with that particular combination of errors and attributes, and unless the software is prepared, chaos can ensue. In general, it is best to assume that the network is filled with malevolent entities that will send in packets designed to have the worst possible effect.

This assumption will lead to suitable protective design, although the most serious problems in the Internet have been caused by unenvisaged mechanisms triggered by low-probability events; mere human malice would never have taken so devious a course!

Adaptability to change must be designed into all levels of Internet host software. As a simple example, consider a protocol specification that contains an enumeration of values for a particular header field—e.g., a type field, a port number, or an error code; this enumeration must be assumed to be incomplete. Thus, if a protocol specification defines four possible error codes, the software must not break when a fifth code shows up.

An undefined code might be logged (see below), but it must not cause a failure.

The second part of the principle is almost as important: software on other hosts may contain deficiencies that make it unwise to exploit legal but obscure protocol features. It is unwise to stray far from the obvious and simple, lest untoward effects result elsewhere. A corollary of this is “watch out for misbehaving hosts”; host software should be prepared, not just to survive other misbehaving hosts, but also to cooperate to limit the amount of disruption such hosts can cause to the shared communication facility. 

Green (agree)

  • It definitely follows that networks designed according to the robustness principle would be optimally interoperable.
  • It is true that sooner or later any system will encounter an incoming packet that could introduce errors.
  • Software should be able to gracefully handle deviations from their expected inputs, because it decreases the likelihood of an error putting the system in a bad state.
  • While you should be liberal in what you accept, it doesn't mean you have to process inputs outside of your specifications--just that this input should not cause an error.

Red (disagree)

  • Errors that are very unlikely do not have an adverse impact on most people using networks, but when they introduce exploits, they could potentially have a VERY adverse impact.
  • Networked systems should not interpret being "liberal" to mean accommodating outdated or insecure protocols.
  • Malicious actors can notice the potential for these low probability events and adapt the mechanisms to make them even worse than they might have been if they were just caused by chance.
  • Designing a network based merely on the Robustness Principle is probably not enough to lead to suitable protective design.

Whitelisting is one important protection strategy in network security. Generally, it is a good idea to define the finite set of behaviors that are acceptable on the network (even if it is a large set). This gives defenders of networks an advantage over attackers, because the attackers generally can't know the expected behaviors of an arbitrary network without first interacting with it. At the same time, every interaction with the network increases the chance that the attacker commits an unexpected behavior, alerting the network to the suspicious activity.

Another way of protecting networks is to set up zones with different access privileges. A policy, which might be enforced by a firewall, ensures that nodes within the zones are only able to perform actions that are explicitly allowed. For example, many networks might have a "demilitarized zone" (DMZ) which is a public-facing part of the network that is isolated from internal zones.

Web and email gateways similarly provide a layer of defense between a network and outside actors that might need to interact with the network. Each of the different gateways is configured to provide a proxy in different ways based on the function of the gateway. For example, a web gateway might be better equipped than a firewall to selectively filter web traffic without blocking entire sites. An email proxy, on the other hand, can be used to block spam and scan outgoing files for data loss prevention (DLP) (e.g. making sure internal secrets are not being sent out on the network).

A layered defense is important, because if an attacker breaches one layer, they ideally still have several others to contend with. At the center--protected by as many layers as possible--is the internal content or system most valuable to the network such as the database or data center.

A security information events manager (SIEM) logs data and traffic across the network and from all devices on the network and performs behavioral analysis on the aggregated logs, for example, to generate automated defense and/or detect or provide key insights during an attack (such as whether a virus is spreading across the network).

Intrusion detection systems (IDS) and intrusion prevention systems (IPS) are more proactive than firewalls. Whereas firewalls might provide protection (e.g. by blocking certain traffic), intrusion detection and prevention systems are configured to detect the attacks and, importantly, identify them. In one example, an IPS might scan for known exploits or viruses to provide a layer of protection against well known attacks for older software that might not be patched. While these systems can provide quick and efficient protection against known attacks, including virtual patches for newly released software fixes that might take awhile to propagate, they generally are not as good at addressing zero day attacks, and they are prone to false positives.

A honeynet is a network that is intentionally set up to lure in an attacker. These systems provide phony information, often at a significant delay, to waste the time of and deceive attackers.

Quarantine is the notion that certain hosts that misbehave on a network are put in a condition where they can't infect any other devices on the network. This strategy is commonly deployed upon first entry into the network. A common example is airport wifi, which requires an authentication step, even after connecting, before a user can fully access the network.

Reputation policy systems collect a list of information about good and/or bad actors, origins, destinations, files, behaviors, etc and uses the information to protect against attacks while also preventing false positives and allowing legitimate traffic through efficiently. Generally, reputation is a "big data solution," meaning that the value of the collected information increases with the amount of aggregated data on any given aspect of a network.

Different network security products offer different combinations of these detection and protection technologies. For example, a firewall has detection mechanisms such as packet filtering, or deep stateful inspections while also offering protection such as policy enforcement, blocking traffic, modifying traffic, translation and/or routing.

One threat is known as a "man in the middle" (MITM). Here, between any given nodes A and B communicating on a network, M is interposed between them, intercepting and possibly changing messages. Interestingly, a MITM can be used in both attacking and protecting networks. In one example, MITMs might be used to intercept HTTP requests by an attacker, who might want to collect information from the traffic or change it, and by a protection system that might want to provide a layer of protection between outside and inside systems.

Reconnaissance is an important aspect of network security. In active reconnaissance, an attacker seeking to attack a network needs to first collect information about a network. A common technique is scanning, which involves trying to connect to as many hosts and services as possible in order to determine which respond in a manner that indicate that they are open to connections (and thus potentially vulnerable to exploit). Some of these scans are performed over long periods of time, making them very difficult to detect. Passive reconnaissance, on the other hand, involves an attacker simply collecting data on a network to see what they can learn about the users.

Another threat is known as spoofing, wherein an attacker pretends to be another network entity to gain an advantage on the network. For example, IP and ARP spoofing can be used to poison ARP databases. In predictive spoofing, TCP sequence numbers are predicted, allowing attackers to shut down connections.

Resource consumption attacks are some of the most well known. These include denial of service (DoS) attacks, which aim to consume resources of the network for an extended period, disabling or slowing down the network for legitimate purposes. Distributed DoS (DDoS) deploy a large number of computers to consume resources of the network asymmetrically (e.g. more harmful to the network than to the individual computers). These attacks are harder to defend against. Scrubbing centers can provide a partial defense by providing a huge bandwidth to redirected traffic from a protected network, only allowing legitimate traffic through to the network.

Network defenders also must contend with software bugs and back doors. Back doors are intentional vulnerabilities (such as built in or default passwords), whereas bugs are unintentional (such as programs unable to handle certain packets).

An important tool in network security is Network Intrusion Prevention Systems (NIIPS), which evaluate packet data along with header data. These systems can be signature-based (e.g. watching for malicious patterns of traffic or application data) or anomaly-based (e.g. context-aware).

Wireshark is a useful tool for collecting information about networks. This program provides the ability to capture packets traveling across a network. Header data of these packets can be analyzed to find information such as IP addresses, DNS information, FTP protocol commands, MAC addresses of routers and devices, and packet sizes. The content of the packets can also be analyzed for information about the organization or people using the network, including passwords, transcripts, and internet service provider information.

Wireshark Example

The following simulation aims to show how Wireshark can be used to gather important information required to secure a network. The simulation involves fictional bad actors Betty and Greg, who are conspiring to steal data from the network. The simulation begins at a point when Betty and Greg are communicating over the network in order to set up a time and place to meet.

The first challenge is determining when the meeting has been set up. It does not take long to find evidence of Betty and Greg's collusion. A series of IRC packets are immediately apparent within the capture. Assembling them into a stream shows a series of messages discussing when to meet up. Unfortunately, the messages containing the necessary information are not legible.

Fig. 4 - Wireshark presents IRC messages between attackers of a network, some of which are encoded or encrypted.

The concealed messages, however, can be extracted into text form and decoded.

Fig. 5 - The encoded messages are extracted from Wireshark and saved as text.

Next, a simple URL/HTML/XML decoding program is used to convert the strings exchanged between the attackers into plaintext format that can be read and interpreted. As a result, we are able to discern that the meeting between Betty and Greg is scheduled for Wednesday at 2pm.

Fig. 6 - The encoded messages are decoded, revealing their contents.

The second challenge involves determining when the meeting will take place by analyzing a second network capture file. Here, a simple search for the string "where" provides a lead. What appears to be an email between Betty and Greg includes the phrase "where should you meet me."

Fig. 7 - Searching the network packets for the string "where" turns up a promising lead.

The contents of this email can be decoded for clearer reading, allowing better visibility into the exchange and providing another key detail. Betty provides a string ("S3cr3tVV34p0n") which she refers to as a "password for where you should meet me." At this point, it's not clear exactly what the password might be used for, but it is clearly important.

Fig. 8 - A password is extracted from an email between Betty and Greg.

Further digging through the network packets turns up another communication between Betty and Greg. The names betty and Greg's apparent handle "D34thM3rch4nt" appear in another chat transcript, during which Betty attempts a file transfer that fails. The name of the file is promising ("r3nd3zv0us"), but we learn that the transmission was blocked. Betty is directed to find another way to transfer the file.

 

Fig. 9 - A transcript compiled by Wireshark showing an attempted file transfer from Betty to Greg.

Almost immediately after the failed transmission, a series of TCP packets are exchanged between Betty's and Greg's IP addresses. The packets evidently are another attempt at transmitting the file, which is extracted using Wireshark. However, the file appears to be encrypted.

Fig. 10 - Wireshark is used to extract an encrypted file transferred across the network.

From before, we know that Betty provided a password ("S3cr3tVV34p0n") to be used by Greg to determine where to meet. Now she has transferred an apparently encrypted file called "r3nd3zv0us" to Greg. Sure enough, running truecrypt on the file using the password results in a directory containing an image and a text file, which reveal the city of the meeting to be Las Vegas.

Fig. 11 - The meeting location was concealed within an encrypted file transferred via TCP from Betty to Greg.

In the third challenge, a co-investigator has hacked Betty's email, resulting in an apparent malware infection. Again, a network packet capture file is analyzed to get to the bottom of what happened. This time, logged events appear, showing processes on the infected computer starting and stopping. Specifically, msimn.exe, a process associated with Microsoft Outlook, starts, followed by Firefox, which opens and then closes in short order.

Fig. 12 - Logs compiled via Wireshark showing a suspicious sequence of events.

Strangely, after Firefox loads, there is only one HTTP request from the infected computer before the browser is closed again. Using Wireshark's export feature, the requested web page can be extracted and studied. Upon close examination of the file, it is evident that is a good candidate for the malicious payload. For one thing, the URL from which the page was downloaded is unusual URL. More importantly, despite being  3,113 bytes, the html file has no content other than a single script. The script includes a variable n that is assigned a huge chunk of data, followed by some code that, for example, might allow the chunk of data in n to be loaded into executable memory and executed.

Fig. 13 - Despite being over 3,000 bytes long, the HTML file contains no content except for a script, making it the likely payload.
