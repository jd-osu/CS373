Week 4 Write-Up

Vulnerability analysis and exploitation concerns examining software for bad code that can be used (exploited) to put the program in a state that allows you to control it in ways the original programmer did not intend. Generally this boils down to finding bugs or taking advantage of misconfiguration such as a weak password.

This is an important area of study, because more and more malware bypasses external protections (at the perimeter of the targeted system) and instead focuses on taking control of a system within a protected network. A major technique for accomplishing this is exploiting vulnerabilities in browsers. In a typical example, the attacker targets the user's browser (e.g. by hacking a website that the user goes to), triggering a condition that allows the attacker to get control over the browser and the user's computer.

A valuable tool for analyzing programs for vulnerabilties is WinDBG, which is a standard debugger for Windows that allows users to set breakpoints and examine the program state at different points in the program's execution. Data can be formatted in a disassembly view to show program instructions in Assembly language. The content of registers and the format/content of the heap and stack can also be examined in detail.

Using WinDBG requires some familiarity with basic computer architecture. For example, a little knowledge of the different registers and their functions combined with the WinDBG tool allows one to identify vulnerabilities and target them to control Windows programs.

Knowledge of program memory is also useful. Each program has different memory regions. Knowledge of the stack/heap can be used to identify memory that the attacker can control and using access to that memory to control the program.

Fig. 1 - WinDBG being used to exploit a vulnerability. Here, the possibility of controlling the eip register has been established, as the value "42424242" has been written onto the eip register.

Memory corruption is a common way of exploiting vulnerabilities. Here, memory is accessed in invalid ways, resulting in undefined behavior that can be exploited. Some techniques for memory corruption include lifetime control, uninitialized memory, array index calculations, and buffer lengths calculations. In one example, the programmer includes in the program code an insecure use of "memcpy", with a user supplied value for length and for value to be copied with no check to ensure that the length is within a maximum capacity for the buffer. Data in memory can then be written over, including data to be supplied to CPU registers in later execution.

stack - The stack is an area where arguments, variables, and return addresses are pushed during execution of a function, including the main function of the program. The ebp register points to the start of the stack frame, and the esp points to the end of the stack frame. Because data is pushed to the stack in a predefined and predictable sequence, the functionality of the program can be explored and knowledge of where certain instructions are and how to control the memory can be learned from studying the stack during execution. Vulnerabilities allow the stack to be controlled by an attacker, and if an attacker can control the stack, they can control the entire state of the program.

heap - The heap is an area of memory accessed by programs executing dynamic memory allocation functionality. Unused sections of the heap can be allocated and then released (e.g. "freed"). One common exploit takes advantage of a "Use-After-Free Vulnerability". Here, the attacker frees an object, replaces an object with one designed by the attacker, positions the shellcode in a section of memory to be returned to after the object is used, and then use the object again.

Exploits comprise a trigger, which invokes a software bug to obtain control of the program, and a payload, which is an action to be performed to obtain control of the program. Typically the payload is known as "shell" code, as it's usually assembly code for executing a shell, allowing access to other data and functionality of the computer.

Fig. 2 - WinDBG has been used to launch Calculator, a common way of proving that control has been obtained in the field of vulnerability analysis.

One stage of developing exploits is known as crash triage, in which it is determined what exactly (e.g. registers, locations of data on the heap and stack) the attacker has control over, which registers contain or point to attacker-controlled data, and whether the attacker has control over critical data such as stack frames. It is also determined where in the program the vulnerability occurs.

In general, in many of these exploits it is the objective to gain control of the eip register. The register points to where in memory the CPU is currently executing. By taking control of eip, an attacker can cause the program to execute other code than originally intended. In one example, the attacker determines, using WinDBG, how many bytes exist between a buffer and a return address on the stack. Once this offset is determined, eip can be overwritten.

Next, the shellcode needs to be put somewhere in memory. In many demonstrations, the Calculator program is used. In this case, the static location of that program in memory would be specified. However, in other examples, a location of a different program might be used. In a linear stack overflow, the data loaded into memory would include an NOP Sled, followed by the shell code, followed by the return address. A more effective technique involves loading filler data, followed by a special data value, followed by the shellcode at the end.

Once the shellcode is loaded into memory, the address needs to be determined. One technique for doing so is called a "trampoline." Here, the program is made to jump to a location, which then jumps to another location. The Assembly language instruction "jmp esp" is used to accomplish this. The hex value "ff e4" is located in memory and the return address is replaced with the location of that instruction, which then causes the program to jump to the value in esp, which, thanks to the predictable manner in which the program uses the stack and to manipulation of the data in the stack, contains the shellcode.

Fig. 3 - WinDBG being used to search for the location of "ff e4", which is a hex value corresponding to the "jmp esp" Assembly language instruction.

Fig. 4 - A sample JavaScript program that exploits a memory copying vulnerability of a sample program. The exploit constructs a string large enough to write outside of the designated buffer and writes the memory location of a jmp esp instruction to the eip register.

Exploits commonly use the heap portion of memory also. The application accesses the heap via an interface (e.g. API) to a heap manager comprising different functions (HeapAlloc, LocalAlloc, GlobalAlloc, malloc). In addition, a front end allocator uses low fragmentation heap (LFH). The application can also use a memory manager via the VirtualAlloc function. Different means of accessing the heap are used depending on the size of the data being inserted, as indicated in Table 2.

LFH creates buckets for a specific data size after 18 allocations of that size. In one example, a loop is developed to make the allocation 18 or more times in order to ensure that the bucket is created. The location of this bucket--specifically the location of the virtual function table for the object, is then determined.
