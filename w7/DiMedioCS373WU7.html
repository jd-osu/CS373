Week 7 Write-Up

Web security is an expansive topic related to network security that focuses on how the web can be used for malicious acts, for example, by operating as a delivery mechanism for malware.

Web attacks generally evolve very quickly. Early attacks included phishing, popup based redirection and script-bombing. Later, phishing was improved, and cross-site scripting and browser exploits became more prevalent. Still later, phishing evolved into spear phishing, and exploits often focused on browser plug-ins. Today, with so much functionality contained within browsers, many attacks are known as "Man in the Browser" (MITB) attacks, with malicious functionality staying entirely contained within the browser and never venturing out into the OS, making them harder to detect and remove.

Modern web technology provides a lot of potential injection points for malicious functions. For example, attacks can focus on de-obfuscated content (content which has been decoded by a browser and rendered), script engines (e.g. for running JavaScript), HTML or DOM trees of pages, or even HTTP or network layer packets.

By far the most vulnerable injection point is at the user-level. Here, social engineering can be used to get users to do something willingly for malicious intent. In these cases, software-based security measures such as hardened or updated browser's or OSes are often not effective. User traits such as impatience or laziness are often used to get the user to click a link, download software, or provide information. Examples of user-level attacks include phishing, SEO poisoning (e.g. getting links to malicious sites into a top 20 results for certain search terms), fake anti-virus software, social media or forum link insertion, and malvertising.

Of these, phishing involves getting information, such as sensitive financial or identification information or credentials such as a passwords. In one example, a user is presented with a fake web form that is made to look like a site or company that is trusted by the user. The form collects the information and, for example, sends it to a malicious actor.

Fake updates and fake anti-virus software is also a common user-level attack. Here, the software mimics well-known security brands or even OS-level UI components in order to deliver malware or even to deliver nothing (but for a price).

In malvertising, malicious actors use advertising networks as a delivery mechanism for malware. These attacks are especially dangerous because they can appear on very high-profile and trusted sites (as advertisements). Attackers seed an advertising network with malicious ads and define a target based on categories such as certain websites, income, or a specific business field. The website then collects some data about a user and shares it with the advertising network, which then puts the malicious ad onto the clean website. These attacks can be expensive for attackers, but they are very efficient.

In another example, "waterholes" are employed. These are sites or resources that are well known to certain targeted users (e.g. software developers). Hacking the resource can then allow attackers to attack the targeted users.

Some defenses to these attacks include URL/domain reputation systems, site certification services, client and gateway anti-virus and anti-malware systems, safe URL shorteners, educating content providers, and educating users.

Just as user-level attacks exploit the behavior of users, browser-level attacks exploit the behavior of web browsers and/or third-party extensions installed within web browsers. These are especially prevalent because of the nature of web browsing today, in which browsers can be left open indefinitely, with much if not all of computer activity occurring within or connected to the browser.

Modern browsers include a lot of security features, including content security policy enforcement such as same origin policies (SOP) and cross origin resource sharing (CORS), URL scheme access rules, OS isolation/sandboxing, redirection restrictions, content handling/sniffing, disruptive script handling, built-in and plug-in URL reputation clients, and even add-on anti-virus/anti-malware functionality.

One interesting point here is the balancing act browsers must play between protecting a system from malicious web content (e.g. via sandboxing) and simultaneously protecting itself from malicious system behavior such as hooking or DLL injections.

Some systems go so far as to provide a sandbox for the already heavily sandboxed browsers. Individual tabs can even execute within miniature virtual machines, with different OSes (technically) operating from tab to tab.

Nevertheless, browser exploits are still common, as browsers fundamentally still have to download, render and execute web content which can easily have malicious code hidden within it. These exploits usually involve a multi-step process of luring a user to a site, which allows the browser to be exploited to then download and execute a malicious payload, which can be any malware, including non-web-centric malware.

Obfuscation is used to hide the malware (such as a malicious script) within legitimate-looking web content. Malware obfuscation techniques include renaming script variables to be meaningless, short or difficult to read text, removing whitespace (resulting in a monolithic block of text), self-generating code, in which code is executed that assembles other code from other sources to be executed, and encryption methods.

In one example, JavaScript code is known to be so complex that even very simple programs can be written in series of complex statements. Malicious JavaScript code can thus be encoded, compressed encrypted or packed, with strings encoded in hex, resulting in code that is hard to understand and/or detect.

Man in the middle (MITM) attacks can be used to intercept and modify traffic in real time. These attacks require the ability for mid-traffic insertion and might include DNS poisoning, WIFI hacking, ARP attacks, rogue or infected routers, roge proxies, among other examples.

In the previously mentioned MITB attacks, browsers, plug-ins and HTML 5 features can be used to infect a browser. In some of these cases, the malware never ventures out of the browser to risk detection by anti-malware software running on the computer.

Other examples include DNS spoofing, clickjacking, SQL injection, and same origin policy attacks.

WebGoat is a useful tool for learning how these attacks work by attacking a website that is specifically designed to showcase various possible vulnerabilities.

Fig. 1 - WebGoat provides a fake target for researching vulnerabilities in websites.

The following demonstrations use WebGoat to explore stored cross-site scripting (XSS), improper error handling, and SQL injection.

In general, XSS attacks involve same origin policies (SOP) for browsers, which require resources from different origins to be isolated from each other. For example, browsers might not allow a page that was downloaded from one domain to call a script from another, different domain. The goal of XSS is to inject client-side script into the user's browser, bypassing SOP.

One of the lessons provided by WebGoat involves stored XSS attacks, in which the attacker gets malicious code to be stored and then served to a targeted user in order to be executed when the user accesses that information.

In the WebGoat example, one user "Tom" inserts a script into an address field for a user account profile. In the illustrated example, the code will pop up an alert message that says "attacked."

Fig. 2 - In this stored XSS demonstration, malicious code is entered into the "Street" field of a user profile input form.

Later, when another user "Jerry" accesses the profile of "Tom" that contains the malicious code, the string containing the code, including <script> HTML tags, is retrieved from the WebGoat database and served to the user, whose browser interprets the HTML tags as legitimate parts of the page and thus executes the script.

Fig. 3 - An alert message shows how the <script> tags included in the string that was retrieved for the "street" field of Tom's profile were actually interpreted and executed by Jerry's browser.

Improper error handling can also result in vulnerabilities. In another WebGoat example, an authentication form submits a POST request containing the username and password. The exact parameters in the request can be examined by intercepting the request using a browser tool such as Tamper Data.

Fig. 4 - The Tamper Data plugin for Firefox has intercepted a post request used for authentication.

Tamper Data can be used to insert different values for the different parameters being submitted. In this example, Tamper Data can only be used to modify the values of the parameters being submitted. Native browser functionality can be used, for example, to modify the parameter name itself. This effectively removes the "Password" parameter from the request (because the server is not expecting the renamed parameter).

Fig. 5 - Using the "Inspect Element" option of Firefox, the password field is renamed to "PasswordX", which will not be recognized by the server-side code.

The renamed password parameter results in improper access to the account provided in the username field.

Fig. 6 - A vulnerability of the WebGoat involving improper error handling results in improper access to a user account.

In SQL injection attacks, on the other hand, attackers take advantage of the extent to which the client side code can interact directly with the database. Databases can accept user data submissions (e.g. via get and post requests) and manage authentication using the database. Websites designed with lax security allow this database functionality to be exploited.

In the WebGoat example, numeric SQL injection is used to retrieve data from a database in an unexpected format. Using Tamper Data, a post request is intercepted and an SQL query contained therein is altered. By appending "OR 1=1" to the end of the search term, we can ensure that every row of data will be returned in every case, even when only one row (matching the search query) was intended.

Fig. 7 - Tamper Data is used to add "OR 1=1" to the end of the station parameter, allowing access to all data rows instead of just one.

As expected, the row for every station is not only returned but rendered on the page, because 1=1 will always evaluate to true.

Fig. 8 - The numeric SQL injection attack has successfully retrieved more data than was intended.

Classification of web content and URLs is an important part of providing protection from malicious web content. This involves analyzing web addresses and content for clues to whether content is malicious. One particular challenge, however, is the huge number of web pages and URLs that exist.

Manual classification using various web research tools is a necessary part of classification, especially for new threats, but it is too slow for analyzing a significant proportion of web content.

Static analysis involves getting a page and scanning it without rendering it. This automated method can scan for malicious features without executing any of them. Similarly, low-interaction human analysis involves looking for malicious features without executing any code. On the other hand, high-interaction human analysis involves observing the malicious behavior on a sandboxed system or virtual environment by retrieving, rendering and executing the content with a real browser, including letting a full OS become compromised.

In lexical URL classification, characteristics of URLs can be used to classify URLs as malicious or not. Non-malicous URLs are more likely to have the "www" prefix, shorter domains using the .com suffix, older domains, consistent registration information with respect to countries, more than one IP address, with consecutive IP addresses, public registration information and MX records. This approach can be combined with host-based classification, which involves researching details of IP addresses, DNS information, domain registration and SSL server certificates.
