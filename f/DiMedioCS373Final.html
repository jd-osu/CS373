Final Write-Up

Hack The Box (HTB) (hackthebox.eu) is an online platform for training people in cyber security, including topics such as web penetration, cryptography, steganography, hacking software, and data forensics, among other examples.

Invitation Code

The first challenge posed by HTB is simply gaining access to the challenges themselves. Users must find a way to obtain an invitation code that will allow them to create an account. There are no clues provided for getting this code. In this way, it seems HTB both filters for a certain skill level of participants but also provides hands-on training from the very beginning.

Fig. 1 - A prompt for an invite code for Hack The Box, along with an invitation to "hack your way in."

My approach to getting a code started out mostly as exploring. I inspected lots of different elements of the invitation code page using developer tools of my browser. It seemed like this approach was a step in the right direction when I noticed that the console, which typically only shows error messages, had an ascii graphic macro.

Fig. 2 - Using developers' tools in the Chrome browser to inspect the HTB invite code page revealed a graphic displayed in the console.

At this point, given my skill level and the number of obvious clues provided (none), I tried random things like reading request and response packets exchanged between the site and my browser and also entering a random "token" code I found in the code for the web page as the invitation code. Not surprisingly, these random attempts were not productive.

However, buried in one of the JavaScript source files for the page was a reference to what could have been a function called "makeInviteCode". The name of this function was part of a string that was passed as an argument to what seemed like an extremely complicated (but completely typically) JavaScript function. It seemed like the function might parse this string in order to process commands.

Fig. 3 - Buried in the page's JavaScript source file was a reference to a function called "makeInviteCode".

I wasn't sure how to call the function, but ultimately I was able to find the actual function in the code using the developers' tools of the browser. I attempted to type in the function name in the console, but nothing happened. I tried again with the trailing "()" that I forgot to include before, and a response packet showed up in the console in response. I expanded it and found a cryptic message: {data: "Va beqre gb trarengr gur vaivgr pbqr, znxr n CBFG erdhrfg gb /ncv/vaivgr/trarengr", enctype: "ROT13"} .

Fig. 4 - Calling the "makeInviteCode()" function results in an encrypted message in the console.

The "enctype" property seemed to be a clue to how to decrypt the message. Some searching revealed that ROT13 was a well known encryption method, and decrypting tools were readily available. Given that confidentiality is not a concern in this simulated challenge situation, I went ahead and used one of the web applications hosted at cryptii.com to decrypt the message, which read: "In order to generate the invite code, make a POST request to /api/invite/generate".

Fig. 5 - Decrypting the message received from the makeInviteCode() function provides the instructions necessary to get the actual invite code.

Using another web application, I sent the POST request as instructed and received a response packet with a "code" property and a "format" property. The format property had the value "encoded".

Fig. 6 - A packet in response to a POST request contains an encoded invite code.

Using a Base64 decoding tool, I was able to get a hyphenated string that looked a lot like an invite code.

Fig. 7 - Decoding the code from the response packet results in an apparent invite code.

Unfortunately, this code did not work. An error message appeared saying that my IP address could not use the invite code entered.

Fig. 8 - An error message rejects the invite code based on IP address.

At first I wasn't sure how to fix this issue. I considered using a proxy, but then it wasn't clear what that would even accomplish. Ultimately, I realized that the request sending the code, because it was from a web application, would have been sent from a different IP address than mine. The invite codes are probably generated based on IP address or otherwise are associated with specific IP addresses. Thus, I had to repeat the process using a tool that would send the request directly from my IP address. Using a command line tool called "curl" (https://curl.haxx.se/), I successfully generated a new encoded invite code and decoded it.

Fig. 9 - Using curl, a proper invite code for the local IP address was generated.

Entering the second code was successful. A congratulations message was displayed along with a registration screen.

Fig. 10 - A registration screen is displayed after entering the correct invite code.

Challenge 1: Took the Byte (20 points)

The first challenge is a forensics challenge in which a password needs to be recovered from a file.

Fig. 11 - HTB's "Took the Byte" challenge asks users to recover a password from a given file.

Downloading and extracting the file provided by HTB using the password ("hackthebox") results in a single generic file with no extension called "password".

Fig. 12 - The "password" file contains data from which the password must be recovered.

Opening the file using a general text editor (such as Notepad++) unsurprisingly results in a jumble of characters without much indication of how to proceed.

Fig. 13 - The "password" file as viewed from a text editor.

More illuminating was opening the file using a hex editor, in this case "HxD". The hex editor shows the sequence of bytes in hex along side the decoded text. By looking at the bytes in hex directly, it is possible to see if there are any patterns in the file that wouldn't be evident from the decoded text, which in this case is meaningless because the file does not contain text data.

Fig. 14 - The bytes of the "password" file as revealed by HxD, a hex editor.

One particularly unusual thing I noticed was how many bytes were "FF", which seems unintuitive. On the contrary, one would expect a data file to contain more whitespace and thus more "00" bytes.

Based on this observation, I attempted inverting each bit of the file (e.g. changing each 1 to 0 and changing each 0 to 1). Because the file was so short, I simply did this manually using HxD. Instead of doing it on a bit level, I mapped each hex digit to its inverted version and then simply typed over the hex digits.

This resulted in a file with legible decoded text.

Fig. 15 - Inverting each bit in the "password" file results in meaningful text data.

From looking at the file, it is clear that we are one step closer to recovering the password. The decoded text includes the string "password.txt". However, other than this reference to a file, there is no actual password.

I noticed that the string "PK" repeats a few times throughout the file. After doing a search, I found a Wikipedia article (https://en.wikipedia.org/wiki/List_of_file_signatures) on file signatures. According to the article, the "PK" is a file signature for .zip files. Thus, the "password" file could itself be a .zip file (which would in turn include the compressed password.txt file).

Indeed, appending ".zip" to the file name results in a zip file. Decompressing the .zip file results in a "password.txt" file.

Fig. 16 - A "password.txt" file was compressed within the "password" file.

As expected, inside the password.txt file is a HTB signature.

Fig. 17 - "password.txt" contains the recovered password and the solution to the challenge.

Submitting the HTB signature confirms that the challenge is solved.

Challenge 2: Unified (20 points)

The second challenge is a steganography challenge in which a file is said to contain a hidden message.

Fig. 18 - HTB's "Unified" challenge asks users to find a hidden message in a file.

Downloading and extracting the file provided by HTB using the password ("hackthebox") results in a text file called "BOD_30079.txt".

Fig. 19 - The "BOD_30079.txt" file contains a hidden message.

Opening the file using a general text editor (such as Notepad++) reveals a seemingly mundane message providing information about the unicode encoding standard.

Fig. 20 - The "BOD_30079.txt" file as viewed from a text editor.

There are some characters from different alphabets (seemingly Chinese, Arabic, Greek and Cyrillic). All of the messages in different languages, however, translate to the same sentence via Google translate: "The system works in many languages."

Fig. 21 - The Chinese text reads, "The system works in many languages."

I wasn't quite sure what to do next. I checked the properties of the file to see if there was anything revealing in there. The file turns out to have a size of 897 bytes but a size on disk of 4.00 KB. I wasn't sure if this was useful information or not.

Fig. 22 - The properties of the "BOD_30079.txt" file shows a smaller size than size on disk.

I went back to the text file and inspected it again. This time, I noticed that hidden within the foreign language letters was a repeating string of symbols that indicated to me that the data was not able to be decoded into intelligible text (and thus that it might contain the hidden message).

Fig. 23 - Upon closer inspection of the text file, a sequence of symbols is buried within the foreign language text.

Using the hex editor (HxD), I was able to pinpoint the location of the symbols (by finding the text before and after the symbols and extracting the data in between). I saved this data into a new file so I could focus on it without the extraneous bytes.

Fig. 24 - The possible hidden message shown as bytes in hex using HxD.

Upon analyzing the bytes, I noticed a pattern. There was a repeating sequence of "F3", "A0", "80"/"81", and then an arbitrary byte. I wasn't sure what to make of this exactly, but it was evidence that this was the correct data and that the data was formatted in some way.

I did a web search for the sequence and came upon a web page with a table of bytes corresponding to different unicode characters (https://www.utf8-chartable.de/unicode-utf8-table.pl?start=917376). I wasn't sure why this data wouldn't encode along with the rest of the file into legible text, but it definitely seemed like each of these four-byte sequences corresponded to a specific character.

I looked up each sequence in the table and manually typed out the character matching the byte sequence.

It took two attempts (in one, I got a character wrong), but sure enough, a HTB signature was the result.

Fig. 25 - Manually decoding the byte sequence using the table results in a HTB signature (in two attempts).

Submitting the HTB signature confirms that the challenge is solved.

Challenge 3: Forest (40 points)

The third challenge is a steganography challenge in which an image file is said to have a hidden message.

Fig. 26 - HTB's "Forest" challenge asks users to find a hidden message in an image.

Downloading and extracting the file provided by HTB using the password ("hackthebox") results in an image file called "forest.jpg".

Fig. 27 - The "forest.jpg" file contains a hidden message.

Opening the file reveals a photograph (or possibly a piece of digital artwork) of a forest. Nothing obvious presents itself as a possible lead.

Fig. 28 - The "forest.jpg" image file contains a picture depicting a forest.

By doing some preliminary research on steganography using image files, I learned (from https://ctfs.github.io/resources/topics/steganography/file-in-image/README.html) that in many cases the hidden message that is hidden in an image file is placed after the image flag, which is ffd9 for .jpg files. Anything after this tag in these files will be ignored when the image is rendered. Based on this lead, I opened up the "forest.jpg" file in the hex editor (HxD). However, there was no .jpg image flag followed by hidden data.

It occurred to me that image might be published on the web. If I could find the original, I might be able to compare it with the version with the hidden message to see how the file had been modified. I did a reverse image search (using Google) and found a published version of the image. In fact, the image seemed to be a popular wallpaper.

Fig. 29 - The "forest.jpg" seems to have been generated based on a popular wallpaper.

The fact that there seemed to be multiple versions of the same image across the internet did not bode well. If any of these differed from each other, it would be hard to determine whether differences between the wallpaper version and the hidden message version were due to the hidden message itself or simply because a different wallpaper version was used to generate the hidden message version. With this possible snag in mind, I went ahead and downloaded one of the wallpaper versions.

Using HxD, I compared the bytes of each file, resulting in a view of the delta between the two images.

Fig. 30 - Using HxD, two versions of the image file were compared to see if the difference between them would provide meaningful information.

Ultimately, however, this comparison was not fruitful. The differences seemed to span the entire file, with no discernible pattern.

I decided to search the web for some steganography tools, particularly those dealing with image files. After experimenting with a few different programs with no luck, I found a web application at https://osric.com/chris/steganography/decode.html that accepts an image file as an input and outputs a decoded message as another image file. I uploaded the file (assuming confidentiality to not be a factor under these simulated challenge conditions) and, finally, a hidden message was revealed in the form of a string that seemed to be overlaid over each pixel.

Fig. 31 - A hidden string was graphically embedded in the pixels of the image file.

After studying this string closely and comparing it with a few fonts (it seemed to be Comic Sans), I determined that the string reads, "IsJuS1Af0r3sTbR0".

I wasn't sure what to make of this code. I entered it as the HTB signature, but it wasn't, which I could have probably guessed ahead of time given that it wasn't in the proper format to be a signature.

Having a password, however, reminded me that several of the steganography tools that I had previously researched required passwords (or at least, to be able to extract the message a message would have had to have been embedded into that file using the password). Previously, I skipped those tools, because I didn't have a password. Now, I went back to see if any of them would be useful at this point.

Using a program called "steghide", I attempted to extract a message and entered the password when prompted. A message was displayed indicating that the extracted message was written to a file called "nothinghere.txt".

Fig. 32 - A hidden message was successfully extracted from the "forest.jpg" file using steghide and the discovered password.

Opening the "nothinghere.txt" file using HxD, I discovered that the hidden message had successfully been extracted. However, it appeared to be encrypted.

Fig. 33 - The hidden message extracted from the image file is still encrypted.

At the end of the encrypted text, I noticed the string "UGO{NzNm1aTfXvyYmMOe0}". This looks like a HTB signature. Knowing that these signatures always begin with "HTB", I had a lead on how to decrypt the code. Sure enough, the characters U, G and O were each 13 characters away from H, T and B respectively.

I discovered through a web search that this is known as a Caesar shift cipher, and tools are readily available for encrypting and decrypting messages. I entered the encrypted message at one of the tools (https://crypto.interactive-maths.com/caesar-shift-cipher.html) and entered "13" as the key. The result was: "HTB{amaz1ngskillzzbr0}".

Fig. 34 - Using a web-hosted tool, the HTB signature is successfully decrypted.

Finally, noticing that the original message had a pattern of uppercase and lowercase characters, I corrected the case of the decrypted message to match the original, resulting in the string: "HTB{AmAz1nGsKilLzZBr0}".

Submitting the HTB signature confirms that the challenge is solved.

(For what it's worth, the full decrypted message extracted from the image file appeared to be some kind of literary quote about forests.)
