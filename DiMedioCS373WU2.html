Week 2 Write-Up

This week covered advanced forensics methods and tools, which might be used in cases of fraud, intellectual property theft, hacker intrusions or data breaches, inappropriate use of the internet (e.g. in a workplace), child exploitation, or for ediscovery in criminal and civil litigation.

In general, in the field of forensics, the methods boil down to evidence acquisition, investigation and analysis, and reporting the results. In the latter case, it was emphasized how our role is not to prove a suspect guilty or not but actually to prove what happened on the system (for example, without distorting the facts with any specific interpretation). Interestingly, despite this limitation to the role of the forensics investigator, certain tools from the non-digital world might prove to be useful including testimony from interrogations, which might even be conducted by the forensic investigator.

In live forensics, evidence is accessed and preserved with minimal disturbance to the system. Specifically, the live forensic work is conducted at the site of the incident with the system in its normal state (e.g. it hasn't been powered down or transported). Here, the most important consideration is to minimize data loss while also ensuring the safety of the incident response team. For example, every action taken on a system in the context of live forensics erases some amount of memory, thus destroying some evidence. In post-mortem based and network forensics, on the other hand, the evidence is gathered after the system has been shut down, for example, from captured memory or disk images, or from network logs.

In the context of forensics, evidence is considered anything that can be used to prove or disprove a fact, including information about the network, operating system, databases and applications, peripherals, removable media, and, as mentioned above, even human testimony.

The integrity of the evidence must be preserved. In one example, a cryptographic hash of a disk image and each partition thereof is computed. After the data is copied and analyzed, a hash of the copy is created. The hash results must match that for the original data, which is stored in a secure place with limited access.

One tool that proves useful in preserving evidence is a writeblocker, which intervenes between a drive and a computer, preventing the computer from writing to or modifying the drive. The data is retrieved from the drive via the writeblocker and then copied to a different drive.

The incident response (IR) process, in more detail, involves preparation of an IR team (which is usually done on an ad hoc basis in most organizations and situations), detection of an incident, the initial response (e.g. killing the network connection for the affected system), formulating a response strategy, collecting data, performing forensic analysis, and documenting findings from the analysis, which might be used to drive legal action, administrative action, remediation and evaluation efforts.

One area where it might be possible to detect incidents is at the point of a firewall between a DMZ and the internet and the firewall between the DMZ and an internal network. These firewalls, for example, might keep logs that can be accessed or monitored. Additionally, disks from the DMZ system that was affected would be retrieved along with those of the affected host systems. In general, most evidence is collected at an installation stage of an APT, with some possibility of collecting evidence at the delivery stage.

Forensic analysis can present a number of challenges, including the sheer amount of data to be analyzed, formulating a timeline (which might involve synching timing information between a number of systems which may or may not be out of synch), and having skills and tools available to the IR team that are applicable to a number of different types of systems that might be affected, which might have, for example, different log formats to be interpreted.

One important consideration is Locard's Exchange Principle, which states that when two objects come in contact, there is evidence left over of the contact. In other words, it is impossible to interact with a live system without having some effect on it.

As a result, an Order of Volatility (set forth in RFC 3227) indicates the order that evidence should be gathered. First system memory should be collected, followed by temporary file systems, process table and network connection, and network routing information, all of which should be performed in a live forensics context. Proceeding to a post-mortem stage, the investigator can then gather evidence from disks, remote logging and monitoring data, physical configuration of the system and network topology, and backup information.

In one example, a hibernation file provides a snapshot of memory which could provide valuable information during an investigation.

The FTK Imager tool can be used to obtain such hibernation files, among other examples, such as memory dumps, reading and copying the disk, exporting the master file table, and obtaining other protected files.

Fig. 2 - FTK Imager being used to read contents of a disk.

Data that is extracted via FTK Imager can also be processed and analyzed via other tools. The Volatility tool is one such example.

Volatility provides memory forensics via a series of plugins which can be used to analyze a memory dump (e.g. generated via FTK imager) from a number of perspectives.

Volatility's "imageinfo" plugin provides general information about the memory dump that can be used in subsequent processing. For example, suggested profiles for the memory format can be provided, which are used in calling the other functions.

Fig. 3 - Volatility's "imageinfo" functionality being used to analyze a sample memory dump generated via FTK Imager during a lab session.

Volatility's "psscan" function provides information about processes represented by the data in the memory dump.

Fig. 4 - Volatility's "psscan" function being used to get process information from a memory dump during a lab session.

In other examples, Volatility's "dlllist" function can be used to determine which DLLs were being accessed by the different processes represented in the memory dump, the "netscan" function can be used to view devices on the network, the "deskscan" function can be used to find information about the processes' use of the GUI, and the "getsids" function can be used to find information about users of the system pertaining to the different processes.

Each of the above mentioned functions of Volatility can be printed to a .txt file for easy access and retention.

Fig. 5 - Volatility's "getsids" function being used to get information about users for the different processes during a lab session. In this example, the results were output to a .txt file.

One important part of Windows to become familiar with is the Registry, which records events such as file access, or whether a USB stick was inserted, and other information such as profiles for users or even internet browsing history in some cases. The Registry can be accessed via the Registry Editor. One use for the Registry is in creating a timeline of an incident, although it should be noted that attackers can modify times included in the registry.

Creating a timeline of an attack can be done using a number of other tools, including Volatility (based on a memory dump), accessing a master file table, or Reg-Ripper.

Fig. 6 - Volatility's "timeliner" function being used to create a timeline during a lab session.

Volatility also includes a master file table parser function (mftparser), which extracts the master file table stored in a memory dump and puts it into a text file, including modified, accessed and created time information.

Data recovery is also an important part of forensics. It is possible to recover deleted files, identify when the files were deleted and to find file fragments. Carving programs carve out deleted files in an image based on file headers found in the data. Photorec is one example of a carving program.

Challenge: DPRK Defector

A challenge was presented in class in which a DPRK defector is caught with a hidden USB stick. The challenge was to investigate the disk and answer key questions.

First, the data was mounted as a drive and a number of files were observed still existing in the data (e.g. not having been deleted).

Among the files was an image of Kim Jong Un, which was examined using Notepad++, revealing a password.

Fig. 7 - Password hidden in image file.

The password was necessary to extract a .zip file on the USB stick containing information about the target of the attack.

Fig. 8 - Target information recovered from .zip file using password.

The .bin file was examined using FileInsight, revealing some instructions related to network activity, including a number of IP addresses.

Fig. 9 - Hidden scripting instructions and IP addresses in .bin file of malware.

Additionally, a number of deleted files were recovered using Photorec.

Fig. 10 - Photorec being used to recover files from an image of the USB stick during the challenge portion of the lab session.

Among the recovered files was an image file that seemed to be related to the attack, demanding a ransom.

Fig. 11 - Recovered image file demanding ransom.

Noting the "SPE" prefix before the data .zip files, it was determined that SPE could pertain to the domain being attacked (e.g. an internal domain for "Sony Pictures Entertainment").

A XOR text search was then performed using FileInsight in the .bin file for the string "SPE", resulting in a list of usernames and passwords.

Fig. 12 - Username/password list recovered from the .bin malware file by performing a XOR text search.

Also among the deleted files was a .exe file, which seemed to be similar to the .bin file. Thus, it was determined that the .bin file could be changed to a .exe file to be executed, allowing the behavior of the malware to be studied.

The .bin malware file was copied and renamed to malware.exe and executed with the tools FakeNet and FlyPaper (blocking program exit only) running, based on the assumption that the malware would attempt network connections. Outgoing network activity was detected via FakeNet.

Fig. 12 - Network activity detected upon executing the malware found on the USB stick.

Based on this detected activity and the instructions found in the .bin file, the malware appears to either open the system up to external access or otherwise send data externally.
